# 联机锄地 · 变体线路制作规则

> 本文档面向**线路制作者**。多人联机锄地时，不同玩家可以跑同一条线路的不同"变体"（例如 A 变体打左边的怪、B 变体打右边的怪），但必须在战斗点附近对齐。要让变体能正常配对，请严格按本规则制作线路。

---

## 一、什么是"变体线路"

一个战斗点附近常有两波怪分布在地形两侧。如果所有人挤在一侧打，另一侧的人要空等。变体机制让：

- **A 变体**：把玩家导向左侧那波怪
- **B 变体**：把玩家导向右侧那波怪
- 两个变体**过程可以不同**（走法、节点数都能不一样），但在战斗点前用相同的"同步点标记"对齐，打完一起继续。

最多支持 4 个变体：A / B / C / D。

---

## 二、目录结构与命名（最重要）

在**总线路文件夹**下建固定命名的变体子文件夹，名字**必须**是这 4 个之一（写死匹配，不能改）：

```
传奇/                         ← 总线路文件夹（房主选这个，同步给所有人）
├── A变体/
│   ├── B001南陵传奇_a.json
│   ├── B002某某线路_a.json
│   └── 变体说明.txt          ← 可选，见第六节
├── B变体/
│   ├── B001南陵传奇_b.json
│   ├── B002某某线路_b.json
│   └── 变体说明.txt
├── C变体/                    ← 可选
└── D变体/                    ← 可选
```

规则要点：

1. **子文件夹名固定**：只能是 `A变体` / `B变体` / `C变体` / `D变体`。其它名字（如 `左路`/`E变体`）不会被识别。
2. **文件名后缀对应文件夹**：A变体里的文件统一加 `_a` 后缀，B变体加 `_b`，C加 `_c`，D加 `_d`。
3. **同一条线路在各变体里"去掉后缀后的基名必须完全一致"**，这是配对的依据：
   - `A变体/B001南陵传奇_a.json` 去掉 `_a` → `B001南陵传奇`
   - `B变体/B001南陵传奇_b.json` 去掉 `_b` → `B001南陵传奇`
   - 基名相同 → 系统认定它们是"同一条线路的 A/B 两个变体"。
4. **基名差一个字都配不上对**。比如一个叫 `B001南陵传奇_a`、另一个叫 `B001南陵传奇1_b`，去后缀后基名不同，会被当成两条不相关的线路。

---

## 三、JSON 里要加的字段：`sync_point_id`（同步点标记）

这是整个机制的核心。普通线路靠"文件名 + 位置索引"自动生成同步点，但变体之间走法不同、节点数不同，位置索引对不上，所以**变体必须用显式的 `sync_point_id` 手动标记同步点**。

### 3.1 怎么标

`sync_point_id` 可以标在**任意一个 waypoint** 上——战斗点（`action: "fight"` 那个节点）本身可以标，战斗点的前一个节点（接近 / 走位节点）也可以标，由你按"希望大家在哪一刻对齐"自己决定：

- 标在**战斗点前一个节点**：大家在准备进战斗之前对齐，再各自冲向自己那侧的怪。常用于希望"开打前先集合"的场景。
- 标在**战斗点本身**：到达战斗点这一刻对齐。

两种都合法，按线路实际需要选。下例演示标在战斗点前一个节点（`id:4`）：

```json
{
  "action": "",
  "action_params": "",
  "id": 4,
  "move_mode": "dash",
  "sync_point_id": "fight_1",
  "type": "path",
  "x": 1980.50,
  "y": 1589.30
},
{
  "action": "fight",
  "action_params": "",
  "id": 5,
  "move_mode": "dash",
  "type": "path",
  "x": 1990.18,
  "y": 1590.61
}
```

> 唯一的硬要求：A/B/C/D 各变体要把同一个 `sync_point_id` 标在**逻辑上对应的位置**（都标战斗点本身，或都标对应战斗点的前一个节点），并保证名字一致、顺序一致（见 3.4）。具体标在哪个节点不限死。

### 3.2 A/B 变体怎么配对

A 变体和 B 变体里，**对应同步点要标相同的 `sync_point_id`**（都标战斗点本身，或都标对应战斗点的前一节点，保持逻辑位置对应即可）：

| 逻辑位置 | A变体（打左） | B变体（打右） |
|---|---|---|
| 第 1 个同步点 | `sync_point_id: "fight_1"` | `sync_point_id: "fight_1"` |
| 第 2 个同步点 | `sync_point_id: "fight_2"` | `sync_point_id: "fight_2"` |

两个变体在 `fight_1` 处生成的同步 ID 一致 → 跑 A 的玩家和跑 B 的玩家会在这里互相等待、一起放行，然后各自冲向自己那侧的怪。

### 3.3 命名建议

- **用字母前缀**，如 `fight_1`、`fight_2`、`pickup_1`。
  - 不要用纯数字（如 `1`、`2`），避免和自动模式的内部命名空间混淆。
- 命名要有意义、可读，方便对照。

### 3.4 硬性约束（违反会被拒绝加载 / 校验失败）

1. **同一条线路（同一个 JSON 文件）内，`sync_point_id` 不能重复**。
   - 例如一个文件里出现两个 `fight_1` → 加载阶段直接报错拒绝运行（`InvalidRouteException`）。
   - 多个 waypoint 不标 `sync_point_id`（留空）是允许的，空值不算重复。

2. **各变体的"同步点列表"必须完全一致：数量相同、顺序相同、名字相同**。
   - A 变体标了 `fight_1, fight_2`，B 变体也必须是 `fight_1, fight_2`，且出现顺序一致。
   - 如果 A 是 `fight_1, fight_2`、B 是 `fight_2, fight_1`（顺序反了）→ 联机开始时校验失败，**直接拒绝执行**。
   - 如果 A 有 2 个、B 只有 1 个 → 同样校验失败。

3. **各变体的"传送点序列"必须完全一致**（见第四节）。

---

## 四、传送点（teleport）规则

线路里每个 `type: "teleport"` 的点都是一个**强制传送同步点**，系统会自动按它在路线里的位置（第几段、段内第几个点）编号。

约束：**各变体的传送点序列必须完全一致**——传送点的数量、每个传送点所在的"段索引/段内索引"位置都要对上。

实操建议：

- A/B 变体的**传送结构尽量保持一致**：相同数量的传送点、相同的分段顺序。
- 通常做法：A、B 从**同一个传送锚点**出发（开头的 teleport 完全一样），分叉只发生在传送之后的路径走法上。这样传送序列天然一致，只需保证战斗点前的 `sync_point_id` 对齐即可。
- 如果你在某个变体里多塞 / 少塞了一个传送点，或者把传送点挪到了不同的分段位置，校验会失败。

> 段的划分：路线从头开始，遇到一个 teleport 且当前段已有点位时，就开启新的一段。第一个点如果是 teleport，它属于第 0 段。制作时不用手动算，只要保证 A/B 的传送点摆放结构一致即可。

## 五、放在哪里 / 怎么被加载

- 变体线路放在锄地一条龙的线路资源目录下，例如内置线路目录：
  `GameTask/AutoHoeing/Assets/<总文件夹>/<X变体>/<文件>.json`
- **房主选总文件夹**（如 `传奇`），照常把线路列表同步给所有成员。
- 系统对每条线路**只跑一个代表**（按 A→B→C→D 取第一个存在该线路的变体），不会把 A、B 都跑一遍。
- 每个玩家在配置里对**这个总文件夹**选一次自己要跑的变体（A/B/C/D），整个文件夹下所有线路都按这个变体跑。成员收到房主的线路后，会自动替换成自己选的变体文件来执行。
- **没选变体 / 选的变体里缺这条线路** → 自动回退跑代表（A 变体），不会报错中断。

### 兼容性

- **老线路（没有变体子文件夹、没有 `sync_point_id`）行为完全不变**，照旧全自动同步。
- 变体机制是纯增量功能，不影响现有线路包。

## 六、变体说明文件 `变体说明.txt`（可选，推荐）

为了让队友在 UI 里选变体时知道每个变体干什么，可以在**每个变体子文件夹**里放一个 `变体说明.txt`：

```
传奇/A变体/变体说明.txt   内容：打左边怪
传奇/B变体/变体说明.txt   内容：打右边怪
```

- 内容写**一行简短说明**，建议 ≤ 10 个字符（系统只取第一非空行，过长会被截断）。
- 选择变体的弹窗里会显示：`A变体（打左边怪）` / `B变体（打右边怪）`；选完后按钮上也会带说明。
- 这个文件不是 `.json`，不会被当成线路执行；没有也不影响功能，只是少了提示文字。

---

## 七、制作 Checklist（提交线路前自查）

- [ ] 变体子文件夹名是 `A变体`/`B变体`/`C变体`/`D变体`（一字不差）
- [ ] 各变体里同一条线路的文件名"去掉 `_a`/`_b`... 后缀后的基名"完全一致
- [ ] 文件后缀和所在文件夹对应（A变体里全是 `_a`）
- [ ] 需要对齐的位置都标了 `sync_point_id`（标战斗点本身或其前一个节点均可），用字母前缀（如 `fight_1`）
- [ ] A/B/C/D 各变体的 `sync_point_id` **数量相同、顺序相同、名字相同**
- [ ] 同一个 JSON 文件内没有重复的 `sync_point_id`
- [ ] 各变体的传送点（teleport）数量与摆放结构一致
- [ ] （可选）每个变体子文件夹放了 `变体说明.txt`，一行简短说明

---

## 八、常见错误对照

| 现象 | 原因 | 解决 |
|---|---|---|
| 变体没出现在选择列表 | 文件夹名不是 `X变体`，或文件没放进变体子文件夹 | 按第二节命名 |
| A、B 被各跑了一遍 | 没放进变体子文件夹（平铺在总文件夹里） | 移进 `A变体`/`B变体` 子文件夹 |
| 联机开始就报校验失败、拒绝执行 | 各变体 `sync_point_id` 数量/顺序/名字不一致，或传送点序列不一致 | 对齐第三、四节的同步点和传送结构 |
| 加载某线路直接报错 | 同一 JSON 内 `sync_point_id` 重复 | 去重 |
| 队友卡在同步点等到超时 | 同步点的 `sync_point_id` 没标，或两变体名字不一致 | 在对应位置标相同的 `sync_point_id` |
| 选变体时看不到说明文字 | 没放 `变体说明.txt` 或内容为空 | 第六节 |
